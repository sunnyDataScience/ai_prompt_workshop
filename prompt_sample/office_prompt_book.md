## 第一章：職場基礎場景

### 場景 1：客戶延遲交貨道歉 Email

### Before -- 許願式提問

```
幫我寫一封道歉信給客戶，說貨會遲到。
```

### After -- 結構化開工單

```
## Task（任務）
撰寫一封客戶道歉 Email，通知交貨延遲。

## Context（上下文）
- 客戶：大中華區經銷商，合作 3 年
- 原訂交期：3/10，實際預計：3/17，延遲 7 天
- 原因：供應商原料短缺（非我方品質問題）
- 我方已安排空運趕工，額外成本由我方吸收
- 寄件人職稱：業務部經理

## Acceptance Criteria（驗收標準）
- 語氣：專業但有誠意，不卑不亢
- 結構：開頭致歉 → 說明原因 → 補救方案 → 後續承諾
- 長度：200-300 字
- 格式：Email 格式，含主旨行
- 禁止：不要推卸責任給供應商、不要過度承諾
```

### 場景 2：社群媒體競品分析報告

### Before -- L0 隨便丟一句

```
幫我做一份 Q2 社群媒體競品分析報告
```

### After -- L1 結構化開工單

```
【S -- 情境】
我是消費品公司的數位行銷經理。這份報告將在週一主管會議
呈報給行銷副總，用來決定 Q3 社群媒體預算分配。

【T -- 任務】
請製作 Q2（2026 年 4-6 月）社群媒體競品分析報告。

【R -- 角色】
你是資深社群媒體分析師，擅長從公開數據推導競爭策略。

【I -- 輸入】
- 我方品牌：[品牌名稱]
- 主要競品：[競品 A]、[競品 B]、[競品 C]
- 分析平台：Instagram、Facebook、TikTok
- 已有數據：（貼上我方 Q2 的貼文數、互動數等）

【K -- 驗收指標】
- 比較維度：發文頻率、互動率、粉絲成長率、內容類型分布、付費推廣估算
- 每個競品用一張表格，最後一張表做總比較
- 結論段：3 個值得學習的做法 + 2 個我方優勢
- 所有數據標註來源（推估請標註「推估」）
- 總長度 800-1200 字

【E -- 範例】
| 指標 | 我方 | 競品 A | 競品 B | 競品 C |
|------|------|--------|--------|--------|
| IG 發文頻率/月 | ... | ... | ... | ... |
```

### 場景 3：會議紀錄的四個等級

#### L0 聊天型的 Prompt

```
幫我整理這份會議紀錄
```

#### L1 結構型的 Prompt（STRIKE）

```
【情境 S】我是專案經理，剛結束一場 1 小時的產品開發週會，
         與會者包含工程、設計、QA 三個部門。

【任務 T】請將以下會議逐字稿整理成正式會議紀錄。

【角色 R】你是一位資深行政助理，擅長從冗長對話中提取關鍵資訊。

【輸入 I】（貼上會議逐字稿）

【驗收 K】會議紀錄必須包含：
         1. 會議基本資訊（日期、時間、與會者）
         2. 討論議題摘要（每個議題 3-5 句）
         3. 決議事項（明確列出）
         4. 待辦事項（含負責人與截止日期）
         5. 總長度不超過 A4 一頁

【範例 E】待辦格式範例：
         - [ ] 完成 API 文件更新｜負責人：王小明｜截止：2026/03/12
```

#### L2 流程型的 Prompt（多步驟鏈式攻擊）

```
=== Step 1：提取原始資訊 ===
請從以下逐字稿中，提取所有「有人提出的議題」和「有人做出的承諾」。
用條列式列出，不要摘要，保留原始語意。
（貼上逐字稿）

=== Step 2：分類整理 ===
請將 Step 1 的結果分類為：
A. 決議事項（已達成共識的）
B. 待辦事項（有人承諾要做的，標註負責人與提到的時間）
C. 未決事項（有討論但沒結論的）
D. 風險/阻礙（有人提出擔憂的）

=== Step 3：生成正式會議紀錄 ===
根據 Step 2 的分類結果，生成正式會議紀錄。
格式要求：（同 L1 的驗收標準）
額外要求：在文末加上「未決事項追蹤表」和「風險追蹤表」。

=== Step 4：品質檢查 ===
請檢查會議紀錄是否符合以下條件：
- 所有待辦都有負責人和截止日？
- 有沒有逐字稿中提到但紀錄遺漏的議題？
- 語氣是否客觀中立？
列出檢查結果和需要修正的地方。
```

#### L3 系統型的 Prompt（系統化設計）

```
=== Agent Spec：會議紀錄處理系統 ===

【系統角色】會議紀錄助理 v2.1
【觸發條件】每次會議結束後，上傳錄音檔或逐字稿
【處理流程】
  1. 語音轉文字（若為錄音檔）→ 呼叫 Whisper API
  2. 資訊提取 → 使用 extraction_prompt_v2.1
  3. 分類整理 → 使用 classification_prompt_v1.3
  4. 生成紀錄 → 使用 formatting_prompt_v2.0
  5. 品質檢查 → 使用 qa_prompt_v1.2
  6. 自動分派 → 待辦事項推送至專案管理工具

【輸出規格】JSON schema：
{
  "meeting_id": "string",
  "date": "ISO 8601",
  "attendees": ["string"],
  "topics": [{"title": "string", "summary": "string"}],
  "decisions": ["string"],
  "action_items": [{"task": "string", "owner": "string", "due": "date"}],
  "open_issues": ["string"],
  "risks": ["string"]
}
```

## 第二章：規格文件與人資應用

### 場景 5：PRD 撰寫

#### Step 1 -- Focus

```
我要一份產品需求文件（PRD），格式為 Markdown，包含：
功能概述、用戶故事、驗收條件、優先級排序、技術限制。
用途：交給開發團隊做下一個 sprint 的 planning。
```

#### Step 2 -- Requirements

```
限制條件：
- 篇幅：不超過 3 頁（Markdown 排版後）
- 受眾：後端工程師 + QA，熟悉技術但不了解商業邏輯
- 語氣：簡潔直接，用條列而非段落
- 格式：每個用戶故事用 Given/When/Then
- 禁止：不可自行編造定價數字或折扣比例
```

#### Step 3 -- Assumptions

```
我的前提假設：
- 會員分為四級：Free / Basic / Pro / Enterprise
- 升降級邏輯基於月消費金額，門檻待定
- 目前系統沒有會員等級欄位，需要新增
- Sprint 週期為 2 週

請在開始之前：
1. 確認你對任務的理解
2. 列出你需要額外知道的資訊
3. 不要自行假設任何商業數字
```

#### Step 4 -- Material

```
我提供以下資料：
- 【附件 1】客戶訪談紀錄摘要（5 位客戶，已整理成重點）
- 【附件 2】現有資料庫 schema（user 表結構）
- 【缺失】競品會員制度比較 → 請標注為 [待驗證]
- 【缺失】財務試算模型 → 不要推測定價
```

#### Step 5 -- Evidence

```
證據規則：
- 來自我附件的資料 → 標注 [附件 1] 或 [附件 2]
- 你的通用知識（如 SaaS 常見做法）→ 標注 [通用知識]
- 推測或不確定的 → 標注 [待驗證]
- 嚴禁編造具體數字
```

#### Step 6 -- Execution Plan

```
請先給我這份 PRD 的執行計畫（大綱 + 每段你打算寫什麼），
不要開始寫正文。
```

#### Step 7 -- Evaluation

```
這份 PRD 的驗收標準：
- [ ] 包含至少 5 個用戶故事
- [ ] 每個用戶故事都有驗收條件（Given/When/Then）
- [ ] 有優先級排序（P0/P1/P2）
- [ ] 技術限制章節不為空
- [ ] 沒有出現 [待驗證] 標記超過 3 處
- [ ] 非技術人員能讀懂（找一個非技術同事試讀）
```

#### Step 8 -- 逐步執行

```
計畫確認。請先完成第 1 段「產品概述」。
完成後停下來，等我確認再繼續。
```

### 場景 8：公文與 Email

#### 30 秒版

```
請幫我寫一封 Email 通知全公司下週一開始實施新差勤制度，語氣正式。
附上制度重點：[貼上重點]
```

#### STRIKE 完整版

```
【S -- 情境】
我是某中型企業的人資行政專員，公司即將於下週一（2026-03-09）
實施新差勤制度。需要發一封全公司公告 Email。

【T -- 任務】
請草擬三個版本的公告 Email：
1. 正式版 -- 適合寄給全公司（含高階主管）
2. 輕鬆版 -- 適合貼在內部 Slack/Teams
3. 風險提醒版 -- 適合寄給部門主管，包含常見爭議點與建議回應話術

【R -- 角色】
你是資深人資溝通顧問，擅長用不同語氣傳達同一訊息，
並能預判員工可能的疑慮。

【I -- 輸入】
新差勤制度重點：
"""
1. 上班打卡時間從 9:00 改為彈性 8:30-9:30
2. 午休時間統一為 12:00-13:00
3. 每月可申請 2 天遠端工作（需主管核准）
4. 遲到超過 30 分鐘需補假單
"""

【K -- 驗收指標】
- 三版各自獨立，可直接複製使用
- 正式版：200-300 字，用「公司」「同仁」等正式用語
- 輕鬆版：150-200 字，可用「大家」「提醒一下」等口語
- 風險提醒版：列出 3 個常見爭議 + 建議回應話術
- 每版結尾都加聯絡窗口與生效日期
- 加入防呆句：「本制度經 [核准單位] 於 [日期] 核定」

【E -- 範例】
正式版格式：
主旨：【公告】新差勤制度自 2026/03/09 起實施
[正文]
聯絡窗口：人資部 [姓名]，分機 [XXX]
```

#### 多步驟版
